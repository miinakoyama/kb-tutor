# Feature Specification: Short-Answer (Constructed-Response) Questions

**Feature Branch**: `002-short-answer-questions`

**Created**: 2026-07-07

**Status**: Draft

**Input**: User description: "Add short-answer question support to the currently MCQ-only app: (1) a student-facing short-answer answering UI with passage highlighting, Part A/B/C sub-questions, up to 2 attempts per part, attempt-history dots, per-attempt feedback blocks, passive unlock countdown, a completion section (key terms + optional notes), a report-feedback modal with teacher review, and a first-time onboarding spotlight tour; (2) an AI item-generation capability based on the no-RAG Method2 (blueprint → item) approach from the mvp4-internal-testing reference project, with generation model selection; (3) AI feedback on submitted answers using three selectable feedback methods (Method 1/2/3) with per-method recommended default models, configurable by teachers and admins."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Student answers a short-answer question and receives AI feedback (Priority: P1)

A student encounters a short-answer question. The screen shows the question context (a stem passage plus a stimulus such as a data table, graph, diagram, scenario, or illustration) in a reading panel on the left, and the sub-questions (Part A, Part B, and usually Part C) as cards on the right. The student can highlight any text in the scenario, questions, or feedback simply by selecting it. Part A is active; later parts are locked. The student types an answer to Part A — the "Check" button becomes clickable as soon as any text is entered — and submits it. While feedback is generated, the textarea and Check button are disabled. Feedback then appears in a colored block with a binary outcome: correct (green) or incorrect (red).

On a first wrong attempt, the student sees supportive Socratic feedback (what's off + a guiding question, never the answer, plus optional glossary term chips) and the textarea re-enables for a second attempt. If the first attempt is correct, the part resolves. On the second attempt, the part resolves regardless of outcome; if still not correct, the student sees a plain-text model answer only (no further hints). When a part resolves, a passive "Part B unlocks in 3…" countdown ticks from 3 to 0 and then automatically unlocks and scrolls to the next part — no button click needed. Two small dots per card track attempt outcomes (green/red) and can be clicked to review that specific attempt. After Part C resolves, a completion section appears with a Key Terms card, an optional lined "My Notes" area, and a Continue button. A first-time viewer is greeted by a short spotlight tour of the real interface, re-openable from a "How to use" button. If the student thinks the AI feedback is wrong or confusing, they can report that part's feedback to their teacher.

**Why this priority**: This is the core value of the feature — without the answering and feedback experience, nothing else (generation, settings) delivers value. It can be demonstrated with a small set of pre-loaded short-answer questions.

**Independent Test**: Seed one short-answer question, log in as a student, answer all parts (including a wrong first attempt and a wrong second attempt), and verify feedback blocks, attempt limits, model-answer reveal, unlock countdown, attempt dots + history modal, highlighting, completion section, feedback reporting, onboarding tour, and that results are saved.

**Acceptance Scenarios**:

1. **Given** a short-answer question with Parts A, B, C, **When** the student opens it, **Then** the stem and stimulus appear in the reading panel, Part A is active, and later parts show a locked state with a "Finish Part A first" note and gray badge.
2. **Given** an empty answer field, **When** the student looks at the Check button, **Then** it is disabled; **When** they type any non-empty text, **Then** Check becomes clickable (no minimum length, no "write more" warning).
3. **Given** a submitted answer, **When** feedback is being generated, **Then** the textarea and Check button are disabled, and they re-enable only if another attempt remains after feedback is shown.
4. **Given** a correct first attempt, **When** feedback appears, **Then** it shows a "Correct!" verdict (green block), no further attempts, and the part resolves.
5. **Given** an incorrect first attempt, **When** feedback appears, **Then** it shows an encouraging verdict ("Good try!" for a full miss, "Good start!" when the response was on the right track), Socratic feedback segments (what's off + a guiding question) with optional glossary chips, a "1 try left" pill, and the textarea re-enables — and the model answer is NOT shown yet. The outcome itself is binary: the block and attempt dot are styled incorrect (red) in both cases.
6. **Given** a second attempt (correct or incorrect), **When** feedback appears, **Then** the part resolves; if not correct, only a plain-text model answer is shown under the verdict ("Here's the idea") with no other feedback segments.
7. **Given** a part resolves (by correct answer or by using both attempts), **When** the feedback settles, **Then** an unlock countdown bar counts "Part B unlocks in 3…" down to 0, then removes itself and unlocks + scrolls to the next part automatically; after Part C it reveals the completion section instead.
8. **Given** a part has scored attempts, **When** the student clicks a colored attempt dot, **Then** an Attempt History modal opens showing only that single attempt: its correct/incorrect pill, the exact text the student wrote, and the feedback they received (segments and, if present, model answer).
9. **Given** the student selects text anywhere in the stimulus panel, a question card, rendered feedback, or the completion section, **When** they release the selection, **Then** the selection is wrapped in a highlight; **When** they click an existing highlight, **Then** it is removed. Selections inside a textarea are never highlighted.
10. **Given** the student submits an empty answer, **When** processed, **Then** the system responds instantly (no AI call) that no response was submitted, and the attempt is counted.
11. **Given** Part C has resolved, **When** the completion section appears, **Then** it shows a Key Terms list (term + one-line definition, all visible), an optional auto-saving "My Notes" textarea, and a Continue button; the bottom bar's "Next" enables and its status reads "All done!".
12. **Given** a first-time viewer of the short-answer screen, **When** the page loads, **Then** a 4-step spotlight tour auto-opens over the real elements (How-to-use button, a highlighted demo word, the Report button, the attempt dots); it can be skipped and later reopened via "How to use", and does not auto-open again on subsequent visits.
13. **Given** any part card, **When** the student clicks Report and sends the report, **Then** the report is recorded for teacher review, the Report button switches to a "Reported" state, and a brief confirmation is shown.
14. **Given** a student who previously wrote notes on one or more questions, **When** they open their notes collection, **Then** they see all their notes listed with question context (topic and a short question preview, newest first) and can open any note to reread it alongside that question; edits made there are saved the same way as in the completion section.

### User Story 2 - Teacher or admin generates short-answer questions with AI (Priority: P2)

A teacher or admin opens the existing bulk question generation screen, which now supports both question types in one run: they select target academic standards and specify how many MCQs and how many short-answer items to generate (either count may be zero). Short-answer counts are distributed across the selected standards the same way MCQ counts are. For short-answer generation they may open an advanced options section to pin a specific core knowledge component (KC) and a stimulus type (table, line graph, bar chart, diagram, scenario, or illustration), and to choose which AI model to use (from Claude Sonnet, Claude Opus, GPT 5.4, GPT 5.4 mini, and Gemini 3.1 Flash Lite) — none of these are required; the system picks valid values automatically. For each short-answer item, the system first produces an item blueprint (which KCs, task types, and stimulus the item will use), then produces the full item from that blueprint: stem, stimulus, 2–3 sub-questions, a holistic rubric, per-part rubrics, and annotated sample responses at every score level. The system validates each item and retries when output is malformed. When the run finishes, all generated questions (MCQs and short-answer items together) are saved into a single question set and the user lands on that set's detail page — the same flow as today's MCQ generation — where each short-answer item can be reviewed in full (rendered stimulus, parts, rubrics, annotated responses) and unwanted items removed with the existing per-question delete.

**Why this priority**: Generation populates the question bank at scale, but the answering experience (US1) can be validated with manually seeded items first.

**Independent Test**: As a teacher (and separately as an admin), run a generation with both MCQ and short-answer counts specified, verify each short-answer item is complete and valid (blueprint, stem, stimulus, parts, rubrics, annotated responses), that the mixed output saves into one question set whose detail page renders both types and supports per-question delete, and that an invalid AI output triggers automatic retry rather than saving a broken item.

**Acceptance Scenarios**:

1. **Given** a teacher or admin on the generation screen, **When** they select standards and specify counts per question type (e.g., 5 MCQs + 2 short-answer) and start generation, **Then** the system produces the requested number of each type and each short-answer item's stimulus type, part task types, and KC coverage match its generated blueprint.
2. **Given** no pinned core KC or stimulus type, **When** short-answer generation runs, **Then** the system selects a valid KC under each item's standard and one of the six supported stimulus types (never "no stimulus").
3. **Given** a saved set containing a generated short-answer item, **When** it is opened on the set detail page, **Then** the stem, rendered stimulus, each part's question, the holistic rubric, per-part rubrics summing to the item total, and annotated sample responses for every score level (0 through full credit) are visible, and the item can be deleted individually like an MCQ.
4. **Given** an AI output that fails validation (missing sections, placeholder text, mismatched stimulus type, rubric points not summing correctly), **When** generation runs, **Then** the system retries that item automatically within a bounded budget and never presents an invalid item as complete; other items in the batch are unaffected.
5. **Given** a saved mixed set, **When** a teacher creates an assignment or a student practices from a linked set, **Then** the short-answer items are available alongside the MCQ items in the same set.
6. **Given** a short-answer count of zero, **When** generation runs, **Then** the behavior of the existing MCQ-only generation is unchanged.

### User Story 3 - Teacher or admin configures the feedback method and model (Priority: P3)

Because the best feedback method has not been decided yet, three feedback methods remain available: Method 1 (single-call grading with strong prompt control and knowledge-base context), Method 2 (two-stage: scoring first, then feedback), and Method 3 (error-analysis-first grading with boundary examples and a confidence rating). A teacher or admin opens the feedback settings for a school (class), chooses which method to use, and optionally overrides the model and temperature. Each method comes with a recommended default (Method 1: Opus 4.8 / temp 1, Method 2: GPT 5.4 / temp 1, Method 3: Sonnet 4.6 / temp 0), which students get unless changed. The setting applies to all short-answer feedback for students in that school. If no school-level setting exists, a system-wide default applies.

**Why this priority**: Students always get feedback via defaults even without this screen; this story adds the ability to compare and steer methods, supporting the eventual decision of a single best method.

**Independent Test**: As a teacher, change a school's feedback method from the default to Method 3, have a student in that school submit an answer, and verify the feedback was produced by Method 3 with its default model; verify students in another school are unaffected.

**Acceptance Scenarios**:

1. **Given** a teacher who belongs to a school, **When** they open feedback settings, **Then** they see the current method and model for that school (or the system default) and can change method, model, and temperature.
2. **Given** a teacher switches the method, **When** they do not touch model or temperature, **Then** those become that method's recommended defaults automatically.
3. **Given** an updated school setting, **When** a student in that school submits their next answer, **Then** the new method/model is used — no re-login required.
4. **Given** a student account, **When** they use the app, **Then** they can never see or change feedback method/model settings.
5. **Given** an admin, **When** they open feedback settings, **Then** they can view and edit any school's settings and the system-wide default.

### User Story 4 - Teacher reviews student reports of AI feedback (Priority: P4)

When a student reports a part's AI feedback as wrong or confusing, that report reaches the teacher. In the teacher dashboard, teachers of a school see a list of feedback reports from their students: who reported it, which question and part, the student's optional note, the student's submitted answer, the AI feedback that was shown, and when it happened. The teacher can open the full context to judge whether the AI feedback was actually wrong, and mark the report as reviewed so the list stays manageable. This both closes the loop with students ("your teacher will review it") and produces evidence for choosing the best feedback method later.

**Why this priority**: Reports are already being recorded for US1; this story adds the review surface. Valuable but not blocking — reports accumulate safely until it ships.

**Independent Test**: As a student, report one part's feedback with a note; as a teacher of that school, verify the report appears with full context (student, question, part, answer, feedback, note), open it, and mark it reviewed; verify it no longer appears among unreviewed reports.

**Acceptance Scenarios**:

1. **Given** a student sent a feedback report, **When** a teacher of that student's school opens the reports view in the teacher dashboard, **Then** the report appears with student name, question/part, timestamp, the student's note (if any), the submitted answer, and the AI feedback that was shown.
2. **Given** a list of reports, **When** the teacher marks one as reviewed, **Then** it moves out of the unreviewed list (but remains accessible in a reviewed history).
3. **Given** teachers of other schools, **When** they open their reports view, **Then** they do not see reports from students outside their schools; admins can see all reports.

### Edge Cases

- **AI service failure or timeout during feedback**: the student's attempt must not be silently consumed. The system shows a retriable error and the attempt count is unchanged until feedback is successfully produced.
- **AI service failure during generation**: the run ends with a clear error message after retries are exhausted; no partial/broken item is saved.
- **Empty or whitespace-only answer**: Check is disabled while empty; if somehow submitted empty, resolved instantly without an AI call as "no response submitted" and counted as an attempt.
- **Exam mode**: short-answer items appear in exam mode with exactly one attempt per part and no feedback of any kind during the exam; grading and feedback for all parts happen after the exam is submitted and are shown in the exam review (see FR-037). The two-attempt Socratic retry loop, attempt dots coloring, glossary chips, model answers, unlock countdown, and the completion section (Key Terms, My Notes) are practice affordances that do not apply during the timed exam, which mimics a real test.
- **Student leaves mid-question (closes tab, navigates away)**: previously submitted attempts and their feedback are preserved and shown when the student returns; a typed-but-unsubmitted draft may be lost.
- **Two-part items**: some items legitimately have only Parts A and B; UI, unlock countdown labels, and scoring adapt (per-part rubric points still sum to the item total).
- **Concurrent setting change**: if a teacher changes the feedback method while a student is mid-question, in-flight submissions complete with the configuration captured at submission time; subsequent submissions use the new setting.
- **Feedback content that is not a clean text string**: the system extracts a usable student-facing message from known keys or falls back to a safe generic message rather than showing raw structured data.
- **Low-confidence grading (Method 3)**: the confidence rating is stored with the attempt so it can be surfaced to teachers later; it is never shown to the student.
- **Highlight spanning multiple elements**: if a selection cannot be wrapped directly, the system still applies the highlight robustly rather than silently failing; highlights never overlap into textareas.
- **Mixed question sets**: a set containing both MCQ and short-answer items presents each in its correct interface within one session, and session progress counts both types.

## Requirements *(mandatory)*

### Functional Requirements

#### Short-answer answering experience (US1)

- **FR-001**: System MUST support a short-answer question type consisting of a stem (context passage), one stimulus (table, line graph, bar chart, diagram, scenario, or illustration), and 2–3 ordered sub-questions (Parts A/B/C), coexisting with the current MCQ type.
- **FR-002**: The answering screen MUST present the stem and stimulus in a reading panel and the parts as answer cards, following the layout and interaction concepts described here while using the app's existing visual design language (colors, shapes, typography); all UI text MUST be in English.
- **FR-003**: The Check (submit) button for a part MUST be disabled only while its answer field is empty; any non-empty text MUST enable it. There MUST be no minimum-length requirement and no "write more" warning. On submit, the field and Check button MUST be disabled while feedback is generated, and re-enabled only if another attempt remains.
- **FR-004**: Each part MUST allow at most 2 attempts, resolving as follows: attempt 1 correct → resolved with correct feedback; attempt 1 incorrect → first-try feedback (Socratic hint/guiding question + optional glossary terms) and a re-enabled field for attempt 2; attempt 2 correct → resolved with correct feedback; attempt 2 incorrect → resolved showing a model answer only. A part always resolves after attempt 2 regardless of correctness.
- **FR-005**: Parts MUST unlock sequentially. When a part resolves, a passive unlock countdown bar MUST count down (e.g., "Part B unlocks in 3…") from 3 to 0 at 1-second intervals, then automatically unlock and smoothly scroll to the next part with no button click; after the final part it MUST reveal the completion section instead. Unlocking MUST update the part's badge from locked/gray to its active style, hide the "Finish Part X first" note, and advance the A/B/C progress stepper (finished step becomes a green check, connector turns green, next step becomes active). This reading pause applies only at part transitions (part resolved → next part / completion section); after a first-attempt miss there is no forced wait — the student may retry as soon as the field re-enables.
- **FR-006**: Each part card MUST show a row of attempt dots (one per attempt slot, default light gray and non-interactive). After an attempt is scored, its dot MUST fill green (correct) or red (incorrect) and become clickable. Clicking a colored dot MUST open an Attempt History modal showing only that single attempt: an outcome pill, the student's exact submitted text, and the feedback received (segments plus model answer if present). Dots and modal MUST include accessible labels for outcome and attempt number.
- **FR-007**: Each submission's feedback MUST render as a bordered/colored block whose color encodes a binary outcome (green correct, red incorrect — no separate "partial" state), containing a verdict row (supportive status glyph + short encouraging phrase, which may vary in wording such as "Good try!" vs "Good start!" while the outcome stays binary; on non-final attempts, a "N try left" pill), feedback segments with small uppercase labels (e.g., "What I noticed", "Try this") and 1–2 sentences that may emphasize key vocabulary, and an optional glossary-chip row. First-attempt misses MUST be structured as Socratic guidance (name what's off + pose a guiding question) and MUST NOT reveal the answer.
- **FR-008**: On a resolving final attempt that is not correct, the block MUST show only a plain-text model answer (full-sentence correct answer/explanation, no card/label/other segments) beneath the verdict. A model answer MUST NOT appear on a non-final attempt.
- **FR-009**: Clicking a glossary chip in feedback MUST show a small glossary popup near the cursor with the term and a one-line definition, dismissed on the next click.
- **FR-010**: Highlighting MUST always be on with no toggle: releasing a non-empty text selection within the stimulus panel, any question card (question text or rendered feedback), or the completion section MUST wrap it in a highlight; clicking an existing highlight MUST remove it (merging text cleanly). Selections inside textareas MUST NOT be highlighted. Highlights MUST persist at least for the duration of work on that question. The only affordance MUST be a passive hint (e.g., "Select to highlight").
- **FR-011**: Empty submissions MUST be resolved instantly without invoking AI, with a fixed "no response was submitted" result, and MUST count as an attempt.
- **FR-012**: Per-part scores, submitted response text, feedback received (segments and/or model answer), attempt numbers, outcome, and timestamps MUST be persisted to the student's history, consistent with how MCQ attempts are recorded (including within assignments).
- **FR-013**: An A/B/C part-progress stepper MUST be visible while answering, and question-level progress MUST integrate with the existing session progress display; the bottom-bar status text MUST reflect the current required action (e.g., "Answer Part B to continue", "All done!").
- **FR-014**: After the final part resolves, a completion section MUST appear containing: (a) a Key Terms card listing each relevant term with a one-line plain-language definition, all visible at a glance (no accordion); (b) an optional "My Notes" lined-paper textarea, clearly marked optional, that auto-saves on blur (debounced) with an unobtrusive "Saved" indicator; and (c) a full-width Continue button that advances to the next question. The bottom-bar "Next" MUST enable when this section appears. Saved notes MUST also be shown pre-filled when the student revisits that question's completion section.
- **FR-015**: Students MUST be able to review all their saved notes later in a "My Notes" collection reachable from their main navigation/dashboard: a list of notes with question context (topic, short question preview, last-edited date, newest first), where opening a note shows it alongside the question it belongs to and allows editing with the same auto-save behavior.
- **FR-016**: Students MUST be able to report a part's feedback to their teacher via a Report control on each part card that opens a modal ("Report feedback — Part X") with a short explanation ("Think the AI feedback is wrong or confusing? Your teacher will review it."), an optional "what seems wrong" text field, and Cancel / Send actions. Sending MUST record the report (student, question, part, attempt context — submitted answer and the feedback shown — plus the optional note and timestamp), switch the button to a "Reported" state, and confirm to the student.
- **FR-017**: Teachers MUST be able to see their schools' feedback reports in the teacher dashboard: a list showing student, question/part, timestamp, note, submitted answer, and the AI feedback shown, filterable to unreviewed reports, with the ability to mark a report as reviewed. Teachers see only reports from their own schools; admins see all reports.
- **FR-018**: The first time a student views the short-answer screen, a spotlight onboarding tour MUST auto-open over the real interface with 4 steps (how-to-use button, a live demo highlight on a word in Part A, the Report button, the attempt dots), with step navigation, skip, and viewport-aware positioning; it MUST be re-openable from a "How to use" control and MUST NOT auto-open on later visits.

#### AI feedback on submissions (US1 + US3)

- **FR-019**: On each non-empty submission, the system MUST produce: a credit result (score for that part), student-facing feedback, and — depending on method — a diagnosed gap and/or a confidence rating; and it MUST supply the data the feedback block needs (verdict category, segments or model answer, optional glossary terms).
- **FR-020**: Three feedback methods MUST be available: Method 1 (single-call grading and feedback with adaptation rules and knowledge-base context), Method 2 (two-stage: score + failure type first, then feedback), and Method 3 (error-analysis-first with boundary examples and confidence output). Behavior of each follows the reference specification preserved with this feature (`reference-pipeline.md`).
- **FR-021**: Feedback for a not-yet-resolved attempt MUST NOT reveal the expected answer; it provides scaffolding (hint, guiding question, or redirection) only.
- **FR-022**: The system MUST record, with every graded attempt, which method, model, and temperature produced the result, plus operational metrics (processing time, token usage, and confidence when available), to support future comparison of methods.
- **FR-023**: If feedback generation fails after automatic retry, the student MUST see a retriable error and their attempt MUST NOT be consumed.

#### Feedback configuration (US3)

- **FR-024**: Feedback method, model, and temperature MUST be configurable per school (class), by teachers of that school and by admins; admins can additionally edit a system-wide default used by schools without their own setting.
- **FR-025**: Each method MUST carry recommended defaults applied automatically when the method is selected: Method 1 → Opus 4.8, temperature 1; Method 2 → GPT 5.4, temperature 1; Method 3 → Sonnet 4.6, temperature 0. Overriding model/temperature per school is allowed.
- **FR-026**: Setting changes MUST take effect for subsequent submissions without requiring students to restart anything.
- **FR-027**: Students MUST NOT be able to view or modify feedback configuration, and configuration data (including AI credentials) MUST never be exposed to student-facing clients.

#### AI item generation (US2)

- **FR-028**: Teachers and admins MUST be able to generate short-answer items from the existing bulk question generation screen, which MUST support specifying per-type counts in one run ("n MCQs + m short-answer items", either count may be zero) against the selected standards, distributing short-answer counts across standards the same way MCQ counts are distributed. Optional short-answer parameters (pinned core KC, stimulus type, generation model, temperature) MUST be available in a collapsed advanced section and never required — omitted values are selected automatically. A run with zero short-answer items MUST behave exactly like today's MCQ-only generation.
- **FR-029**: The generation model MUST be selectable from at least: Claude Sonnet, Claude Opus, GPT 5.4, GPT 5.4 mini, and Gemini 3.1 Flash Lite.
- **FR-030**: Generation MUST follow the two-stage no-study-guide approach: first produce an item blueprint (KC selection, part task sequence, stimulus plan, expected response elements), then produce the final item from the validated blueprint. Study-guide retrieval MUST NOT be used in either stage. Details follow `reference-pipeline.md`.
- **FR-031**: Generation MUST be grounded in curated reference data: the KC table for the standard, KC vocabulary, task-type taxonomy, existing item cards (as style priors only), whole-item exemplars, and rubric anchors.
- **FR-032**: When no core KC is pinned, the system MUST select a valid KC under the standard; when no stimulus type is pinned, it MUST select one of the six supported types (never "none").
- **FR-033**: Every generated item MUST include: stem, stimulus asset appropriate to its type, 2–3 parts each asking exactly one thing, a holistic rubric over the item total, per-part rubrics whose points sum to the item total, and annotated sample responses covering every score level from 0 to full credit.
- **FR-034**: Generated content MUST be validated before acceptance (structural completeness, valid KC/task-type/stimulus references, rubric point sums, presence of all annotated score levels, no unresolved placeholder text, stimulus asset matching the blueprint's stimulus type), with automatic regeneration on failure and a bounded retry limit.
- **FR-035**: Stimuli MUST render in a plain black/white/gray worksheet style without decorative color; diagram stimuli MUST be safe to render (no executable content).
- **FR-036**: A generation run MUST save its output — MCQs and short-answer items together — into a single question set and take the user to that set's detail page, matching the existing MCQ flow (no separate pre-save preview step). The set detail page MUST render each short-answer item in full (rendered stimulus, parts, holistic and per-part rubrics, annotated responses) and MUST support deleting individual short-answer items with the existing per-question delete, so unwanted items can be removed after review. Saved sets are available through existing set-linking, assignment, and practice flows. Each saved short-answer item MUST retain its blueprint and generation metadata (method variant, model, grounding summary). A failure while generating one short-answer item MUST NOT discard other successfully generated items in the batch — the run saves the successful items and reports the failures.

#### Cross-mode behavior

- **FR-037**: Short-answer items MUST be available in all student experiences (self/adaptive practice, review, exam, and assignments). In practice/review/assignment contexts, feedback is shown immediately per the two-attempt loop above. In exam mode, the experience MUST mimic a real test: each part allows exactly one attempt, no feedback of any kind (verdicts, feedback blocks, attempt-dot coloring, glossary, model answers) is shown during the exam, and the completion section (Key Terms, My Notes, unlock countdown) does not appear — students simply answer and move on. All parts are graded after the exam is submitted, and scores and feedback (including model answers for incorrect parts) become visible in the exam review.

#### Reference data

- **FR-038**: The curated reference data required by generation and feedback (KC tables, taxonomy, item cards, exemplars, rubric anchors, adaptation rules, knowledge-base content, boundary examples) MUST be bundled with the application and sourced from the mvp4-internal-testing reference project.

### Key Entities

- **Short-Answer Question (Item)**: A constructed-response question: stem, one typed stimulus, 2–3 parts (each with prompt, task type, maximum score, scoring guidance), holistic rubric, per-part rubrics, annotated sample responses (source of the model answer), key-terms/glossary entries, standard/KC metadata, and (for generated items) its blueprint and generation metadata. Lives in question sets alongside MCQs.
- **Item Blueprint**: The intermediate generation plan: target standard, core/anchor KC, per-part KC + task type + function, stimulus type, cognitive demand, expected response elements, common incomplete responses.
- **Part Attempt**: One submission by a student for one part: response text, attempt number (1 or 2; always 1 in exam mode), score awarded, binary outcome (correct/incorrect), feedback (segments and/or model answer), diagnosed gap, confidence (when available), method/model/temperature used, operational metrics, timestamp. Linked to student, question, part, and (when applicable) assignment.
- **Feedback Report**: A student report that a part's feedback seems wrong or confusing: student, question, part, the submitted answer and feedback shown at report time, optional note, timestamp, and reviewed/unreviewed state; surfaced to teachers of that student's schools (and admins) in the teacher dashboard.
- **Student Note**: The optional free-text note a student writes in a question's completion section; auto-saved, scoped to student + question, and browsable later in the student's "My Notes" collection.
- **Passage Highlight**: A student-created marking on question text/stimulus/feedback, scoped to that student and question, persisting at least for the session.
- **Feedback Configuration**: Per-school setting (chosen method, model, temperature) plus one system-wide default. Editable by teachers (own schools) and admins (all schools + default).
- **Reference Knowledge Data**: Curated data powering generation and Method 1 feedback: KC table with vocabulary, task-type taxonomy, item cards, whole-item exemplars, rubric anchors, adaptation rules, knowledge-base contexts, boundary example pairs.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: A student can complete a 3-part short-answer question — reading, answering, and receiving feedback on every part — in a single sitting without assistance, with each part's feedback appearing within 15 seconds of submission in at least 95% of cases.
- **SC-002**: 100% of non-empty submissions receive a score and student-facing feedback; empty submissions resolve instantly (under 1 second).
- **SC-003**: At least 90% of generation runs yield a fully valid item within the automatic retry budget; 0 invalid items are ever saved to a question set.
- **SC-004**: A teacher can find and change their school's feedback method in under 1 minute, and the change is reflected in the very next student submission for that school.
- **SC-005**: Every graded attempt is stored with its method, model, and metrics, enabling a complete method-comparison report over any date range with no missing records.
- **SC-006**: Students can correctly identify how many attempts they have left, understand the attempt dots, and review any past attempt after only the onboarding tour (verified by usability check with at least 5 students or proxies).
- **SC-007**: Existing MCQ practice, exam, review, and assignment flows continue to work unchanged (no regression in current test suite).
- **SC-008**: In exam mode, no scoring or feedback for short-answer items is revealed before the exam is submitted (verified: 0 leaks across a full exam run).

## Assumptions

- **Clarifications resolved**: (Q1) Short-answer items appear in all modes; exam mode allows exactly one attempt per part and defers all grading/feedback until after exam submission (FR-037). (Q2) Two-attempt resolution follows the exact loop in FR-004/FR-008: the model answer is revealed only on a non-correct second (final) attempt, as plain text with no extra segments. (Q3) Both teachers and admins can run generation (FR-028), matching current teacher access to content generation.
- **Binary outcomes, no "partial" state**: Since each part is typically worth 1 point, attempt outcomes are binary (correct/incorrect); there is no amber "partial" block or dot color. Encouraging verdict wording may still vary ("Good try!" vs "Good start!") based on what the feedback method diagnosed, without changing the binary outcome.
- **"Report" wording**: The student-facing control for questioning AI feedback is labeled "Report" (e.g., "Report" button, "Report feedback — Part X" modal, "Reported" state) rather than "Flag". Rationale: "flag" already means "mark a question to revisit" elsewhere in this app (exam mode), so reusing it for a different action would be confusing, and "report" states the intent plainly.
- **Reading-pause placement**: The forced ~3-second pause exists only at part transitions (the unlock countdown in FR-005), not after every feedback display. Rationale: after a first-attempt miss the student must read the feedback anyway to revise their answer — the retry itself forces engagement, and a lockout there would only add friction. The place students tend to rush past feedback is when a part resolves and they move on, so that is where the passive countdown sits.
- **TELeR labeling**: Per this project's level definitions (L2 = no study-guide details and no expected response elements; L3 = study-guide grounding but no expected response elements; L4 = study-guide + expected response elements + common incomplete responses), the no-RAG blueprint variant does not sit cleanly on any single level: it omits study-guide grounding (L2-like on that axis) but still feeds expected response elements and common incomplete responses into item generation via the blueprint (L4-like on that axis). Decision: keep the reference project's identifier for this exact configuration (`method2_blueprint_no_rag_l3`) so results remain comparable with the reference project's runs, and treat the actual grounding configuration flags (blueprint on, study-guide RAG off) — not the level suffix — as the source of truth. If study-guide RAG is added later, it becomes a new variant label rather than a change to this one.
- **Reference project access**: The reference repository (cocoj1115/mvp4-internal-testing) is publicly reachable; its data files (KC tables, taxonomy, item cards, exemplars, rubric anchors, adaptation rules, KB content, boundary examples) and method logic can be pulled directly during implementation, including from non-main branches (`feat/aig-generation`, `method-1`, `method-2`, `method-3`) where needed. No manual file drop-off from the user is expected. (Branch list verified at spec time.)
- **Detailed pipeline reference**: The generation pipeline and three feedback methods are preserved at `specs/002-short-answer-questions/reference-pipeline.md` and are the authoritative behavioral reference for planning; this spec keeps capability-level statements.
- **Configuration scope decision**: Feedback configuration is per school (class) with a system-wide default fallback. Rationale: schools are the unit already grouping students, teachers, question-set links, and assignments; per-assignment configuration would fragment comparison data, and per-teacher configuration would be ambiguous for multi-teacher schools. Per-assignment overrides can be added later if comparison requires them.
- **Attempt-history presentation**: Per the user's detailed spec, the design uses 2 attempt dots per card (gray → green/red once scored, clickable to open a single-attempt history modal). Accessible labels accompany the color coding so outcome is not conveyed by color alone.
- **Visual design**: Layout and interaction follow the user's detailed description and mockups; colors, shapes, and component styling follow the existing app design system (green/forest primary palette). All UI text is in English.
- **Attempts and scoring**: Each part is worth a small integer score (typically 1 point; total typically 3). "Correct" (full credit) on a part resolves it successfully; item results feed existing progress/analytics as per-part scores.
- **Model answer source**: The model answer shown on a failed final attempt comes from the item's annotated full-credit sample response (the score-3 annotated response), using the portion that addresses the failed part; key terms come from the item's KC vocabulary/glossary data.
- **New AI providers**: The app currently uses one AI provider; feedback and generation additionally require credentials for two more providers (Claude and GPT families). These are server-side secrets, never exposed to clients.
- **Persistence details**: Submitted attempts, feedback, feedback reports, and student notes are durably stored; unsubmitted draft text is session-only; highlights persist at least for the session (durable highlight storage is a nice-to-have, not required). The first-time-tour-seen state is stored per student.
- **Generation cost control**: Generation is an on-demand, user-triggered action (not scheduled/bulk-automated here), so per-run model choice and a bounded per-run short-answer count are sufficient cost control.
- **Unified generation screen (clarified)**: Short-answer generation lives in the existing bulk generation screen with per-type counts, not a separate page. Rationale: the only required user input for a short-answer item is the standard — the core KC and stimulus type are auto-selected when not pinned — so the user-facing inputs match MCQ generation, and the user wants mixed-type sets created in one run. The MCQ pipeline is invoked unchanged (one batched call) while short-answer items are generated one at a time (blueprint → item per item); the run merges both outputs into one set. Model/temperature selection applies to short-answer generation only; MCQ generation keeps its existing Gemini pipeline. Review also matches the existing flow: the run saves immediately and redirects to the set detail page (no pre-save preview step); unwanted items are removed there via the existing per-question delete.
- **Deferred (explicit decisions)**: (a) Assigning KCs to MCQ items is out of scope for this feature — the JSONB question payload can gain an optional KC field later without migration; (b) refactoring MCQ generation to one-LLM-call-per-question is out of scope — the run orchestration treats each type's generator as a separate function so that swap can happen later in isolation.
- **Out of scope**: A method-comparison dashboard (data is captured but not visualized here); manual short-answer authoring/editing UI (items enter the bank via AI generation with preview only). These may be follow-up features. The teacher-facing report review view (US4/FR-017) and the student "My Notes" collection (FR-015) are IN scope.
