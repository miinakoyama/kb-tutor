# Implementation Plan: Dark Mode

## Summary

Add app-wide dark mode with three appearance options (**System**, **Light**, **Dark**) controlled from the existing Settings page. **System** is the default and follows the device `prefers-color-scheme` preference, updating live when the OS theme changes.

Technical approach: extend CSS custom properties in `globals.css` with a `.dark` override, persist `appearance_mode` on `user_settings` (mirroring timezone/TTS hybrid localStorage + Supabase sync), apply theme via a `dark` class on `<html>` with an inline FOUC-prevention script and a client `ThemeProvider`, then migrate ~70 component files from hardcoded light-only colors to semantic tokens.

## Technical Context

**Language/Version**: TypeScript 5.x, Node.js 22.x  
**Primary Dependencies**: Next.js 16.1 (App Router), React 19, Tailwind CSS v4, Supabase (`@supabase/ssr`)  
**Storage**: PostgreSQL `user_settings.appearance_mode` + browser `localStorage`  
**Testing**: Vitest + jsdom (unit tests for `appearance-settings.ts`)  
**Target Platform**: Web — Chromebooks, desktop/mobile browsers (≥360px)  
**Project Type**: Single Next.js web application  
**Performance Goals**: Theme switch visible ≤1s; no new hot-path API routes; inline script <1KB  
**Constraints**: SSR-safe localStorage guards; WCAG 2.1 AA contrast both themes; English UI strings only  
**Scale/Scope**: All roles, all pages (~70 TSX files with hardcoded light palette), 1 DB migration

### Implementation phases (for `/speckit-tasks`)

#### Phase A — Foundation (P1)

1. DB migration: `user_settings.appearance_mode`
2. `src/lib/appearance-settings.ts` + tests
3. Dark token definitions in `globals.css` + Tailwind dark variant
4. `ThemeProvider` + inline script in `layout.tsx`
5. Wire `MigrationBootstrap` one-time sync

#### Phase B — Settings UI (P1)

1. Appearance section on `/settings` (segmented System/Light/Dark)
2. Instant theme apply without reload
3. DB sync on change

#### Phase C — Surface migration (P1)

Migrate components from hardcoded light colors to semantic tokens, in order:

1. Shell (`layout`, `AppChrome`, `Sidebar`)
2. Login pages
3. Student flows (home, practice, review, exam, progress, bookmarks, notifications)
4. Teacher (dashboard, assignments)
5. Admin (content, data-analysis)
6. Shared components, modals, diagrams, Recharts

#### Phase D — QA & polish (P2)

1. KaTeX/Recharts dark contrast fixes
2. Manual QA per `quickstart.md`
3. `npm run lint` + `npm test`

### Capacity note (Constitution III)

- **New DB writes**: One upsert per appearance change (same as timezone) — negligible.
- **New reads**: One select on Settings load / migration bootstrap — off hot student answer path.
- **Client work**: O(1) class toggle; media listener only when mode is `system`.
- **Build**: No impact on SSR data fetching; inline script adds ~500 bytes to HTML shell.

## Risks & Mitigations

| Risk | Mitigation |
|------|------------|
| Incomplete color migration leaves unreadable screens | Batch migration with quickstart smoke checklist per surface |
| FOUC on slow devices | Inline script before body content |
| Hydration mismatch | `suppressHydrationWarning` already on `<html>`; provider reconciles on mount |
| Chart/diagram hardcoded colors | Dedicated sub-task for Recharts props + diagram CSS |
